## 四、確認的關鍵架構決策

### 4.1 身分驗證
- **統一使用 `SchoolPortalLib`**：`issueToken()`、`verifyToken()`、`getUser()`
- **管理者識別**：擴充現有 `systemAccess` JSON，加入 `training_admin: true`
  - 範例：`{"library":true,"property":true,"training_admin":true}`
  - 由教職員名冊管理者在名冊試算表手動設定
- **Token 規格**：CacheService，6 小時 TTL（與整合平台一致）

### 4.2 資料庫（Google Sheets）

### 4.3 Google Drive 資料夾結構
```
研習證明根目錄/（ID 填入常數：TRAINING_DRIVE_ROOT_FOLDER_ID）
├── 2025/
│   ├── t0001/        ← 個人資料夾：管理者讀寫，本人唯讀
│   │   └── 2025_t0001_研習名稱_20250601.pdf
│   └── …
└── 2026/
    └── …
```

**檔案命名規則**：`{年份}_{txxxx}_{課程名稱}_{研習日期YYYYMMDD}.{副檔名}`
> 課程名稱超過 20 字時截斷；特殊字元（`/\?%*:|"<>`）統一替換為底線

**個人資料夾授權**：後端呼叫 `userFolder.addViewer(teacherEmail)` 授予本人唯讀

### 4.4 檔案上傳機制
- **方案**：前端 FileReader → Base64 → `google.script.run`（單次上傳）
- **前端限制**：8MB（base64 後約 10.6MB，在 GAS payload 上限內）
- **允許格式**：PDF、JPG、PNG、GIF
- **不實作分塊上傳**：研習證明通常 < 5MB；超過 8MB 顯示友善錯誤提示

### 4.5 API 路由規範
所有請求統一進入 `doPost(e)` → `handleRequest(e)`，action 採版本前綴：

| action | 說明 | 權限層級 |
|---|---|---|
| `v1/getCatalog` | 取得 ACTIVE 研習目錄 | Level 1 |
| `v1/getMyRecords` | 取得自己的登錄紀錄 | Level 1 |
| `v1/submitRecord` | 送出研習登錄（含 base64 檔案） | Level 1 |
| `v1/deleteRecord` | 刪除自己的 PENDING 紀錄 | Level 1 |
| `v1/admin/addCatalog` | 新增研習課程 | Level 2 |
| `v1/admin/editCatalog` | 編輯研習課程 | Level 2 |
| `v1/admin/archiveCatalog` | 封存研習課程 | Level 2 |
| `v1/admin/getPendingReviews` | 取得待審清單 | Level 2 |
| `v1/admin/reviewRecord` | 核准 / 退件（退件需填原因） | Level 2 |
| `v1/admin/getFileUrl` | 取得證明檔案暫時 URL | Level 2 |
| `v1/admin/exportRecords` | 依條件匯出研習紀錄 CSV | Level 2 |

### 4.6 Hub 試算表擴充
- 新增 `TrainingStats` 工作表：儲存各教師本學年 APPROVED 時數摘要
- 每晚定時觸發器同步（非即時），供主門戶儀表板查詢
- **不在即時借還流程中寫入**，避免造成 LockService 競爭
- **提供主門戶儀表板第七個 KPI「年度研習人時彙整」**：`Sync.gs` 的 `syncTrainingStats()` 負責按處室彙整全校 APPROVED 記錄，以一次 `setValues` 批次寫入 Hub.TrainingStats

`TrainingStats` 工作表欄位（Hub 試算表）：

| 中文標題 | 英文 Key | 說明 |
|---|---|---|
| 教師 ID | `userId` | `txxxx` |
| 所屬處室 | `department` | 從 Hub.UserStatusCache 取得 |
| 本學年核准時數 | `approvedHours` | 僅計 `APPROVED` 紀錄 |
| 必修達成數 | `requiredCompleted` | `isRequired=TRUE` 且 `APPROVED` 的筆數 |
| 最後同步時間 | `syncedAt` | Timestamp |

## 四之一、與整合平台的資料模型對應

本系統在跨系統 ER 圖中的定位：

| 實體 | 本系統欄位 | 對應系統 | 對應欄位 |
|---|---|---|---|
| **使用者** | `userId (txxxx)` | 教職員名冊 | `userId (txxxx)`（主鍵） |
| **使用者部門** | 不儲存，動態查詢 | Hub.UserStatusCache | `department` |
| **研習課程** | `catalogId`、`title`、`hours` | 本系統獨立維護（無對應資產） | ─ |
| **登錄紀錄** | `recordId`、`trainingDate`、`status` | ─（不涉及設備借用） | ─ |
| **研習統計** | `approvedHours`（同步至 Hub） | 主門戶儀表板 | Hub.TrainingStats |

**本系統與其他子系統的關係**：
- **唯讀依賴** Hub.UserStatusCache → 取得教師部門資訊（供統計彙整使用）
- **單向寫入** Hub.TrainingStats → 提供儀表板 KPI 資料來源
- **無設備借用關聯**：研習登錄不涉及資產/設備，不接入 Hub.EquipmentRegistry
- **身分驗證**：透過 SchoolPortalLib 共用，不直接讀取教職員名冊試算表

## 五、UI/UX 設計規範

### 5.1 教師端審核狀態顏色
| 狀態 | 背景色 | 左邊框色 | 語意 |
|---|---|---|---|
| `PENDING` | `#FFF3CD` | `#FFC107` | 警告橘：等待中 |
| `APPROVED` | `#D4EDDA` | `#28A745` | 確認綠：通過 |
| `REJECTED` | `#F8D7DA` | `#DC3545` | 危險紅：需處理 |

### 5.2 REJECTED 紀錄操作
- 顯示退件原因文字（必顯示，不可隱藏）
- 提供「重新送出」按鈕，點後跳至送出頁並預填原有課程資料
- 原 REJECTED 紀錄保留（不刪除），新送出建立新 recordId

### 5.3 管理者批次審核
- 預設顯示 PENDING 清單
- 每筆紀錄卡片直接嵌入 [核准] / [退件] 按鈕（不需進詳情頁）
- 退件原因以 Modal 輸入，`minlength="10"` 防止敷衍填寫
- 可勾選多筆後「批次核准」

## 九、開發規範提醒（每次開工確認）

- 輸出語言：繁體中文（台灣慣用語）
- 試算表讀寫：「一次讀取 → 記憶體處理 → 一次 setValues 寫入」，禁止在迴圈內 setValue
- 前端 API：禁止 `fetch()`，全部使用 `google.script.run` 封裝的 `api()` 函式
- Inline event：所有前端函式須在 `<script>` 末尾掛載至 `window.xxx = xxx`
- CSS 斷點：桌面端用 `min-width: 1024px`（GAS 沙箱 viewport 問題，勿用 768px）
- 日期建構：一律用 `new Date(year, monthIndex, day)`，禁用 `new Date("YYYY/M/D")`
- 跨系統查詢：透過 Hub 試算表中介，禁止直接 `openById` 其他子系統試算表
- 個資保護：證明檔案僅管理者可讀寫，本人唯讀；禁止公開連結
- REJECTED 紀錄：保留歷史，不覆蓋；重新送出建立新 recordId
