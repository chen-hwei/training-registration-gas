## 四、確認的關鍵架構決策

### 4.1 身分驗證
- **統一使用 `SchoolPortalLib`**：`issueToken()`、`verifyToken()`、`getUser()`
- **管理者識別**：擴充現有 `systemAccess` JSON，加入 `training_admin: true`
  - 範例：`{"library":true,"property":true,"training_admin":true}`
  - 由教職員名冊管理者在名冊試算表手動設定
- **Token 規格**：CacheService，6 小時 TTL（與整合平台一致）
- **`Session.getActiveUser().getEmail()` 輔助核對**（補充，非取代 txxxx）：當 GAS 部署模式設定為「以存取 Web App 的使用者身分執行」且限定組織內帳號時，可在後端呼叫 `Session.getActiveUser().getEmail()` 取得教師的 `@zlsh.tp.edu.tw` 帳號，並與 `SchoolPortalLib.getUser(txxxx).email` 交叉比對，防止同一台電腦不同老師借用 Token 上傳他人資料。若兩者不符則拒絕寫入並要求重新登入。
  > ⚠️ 此模式下 GAS 執行身分會切換為**教師帳號**，Drive 操作可能受到教師帳號的配額限制；如需 Admin 層級的 Drive 權限，需評估改用 Service Account 或維持「以我的身分執行」部署模式。待 Phase 0 確認部署設定後再決定是否啟用。

### 4.4 檔案上傳機制
- **方案**：前端 FileReader → Base64 → `google.script.run`（單次上傳）
- **前端限制**：8MB（base64 後約 10.6MB，在 GAS payload 上限內）
- **允許格式**：PDF、JPG、PNG、GIF
- **不實作分塊上傳**：研習證明通常 < 5MB；超過 8MB 顯示友善錯誤提示

#### 4.4.1 前端圖片壓縮（⚠️ 重要，必須實作）

老師習慣以手機直接拍攝研習證明，現代手機照片原始大小常達 5–15MB。直接轉 Base64 後 payload 可能超過 GAS 上限並引發逾時。**必須**在 `config.html` 的工具函式中加入 Canvas 壓縮邏輯，在轉 Base64 之前先壓縮圖片。

**壓縮規則**：
- PDF：不壓縮（直接轉 Base64）
- JPG / PNG / GIF：先壓縮至 **2MB 以下**再轉 Base64（目標大小為原始大小的約 1/4）

```javascript
// config.html — 圖片壓縮工具函式（在 prepareFileForUpload 內呼叫）
async function _compressImage(file) {
  const TARGET_BYTES = 2 * 1024 * 1024; // 2MB
  if (file.size <= TARGET_BYTES) return file; // 已夠小，跳過

  return new Promise((resolve) => {
    const img = new Image();
    const url = URL.createObjectURL(file);

    img.onload = () => {
      const canvas = document.createElement('canvas');
      // 等比縮放：面積壓縮比 = 目標大小 / 原始大小
      const ratio = Math.sqrt(TARGET_BYTES / file.size);
      canvas.width  = Math.floor(img.width  * ratio);
      canvas.height = Math.floor(img.height * ratio);

      canvas.getContext('2d').drawImage(img, 0, 0, canvas.width, canvas.height);
      URL.revokeObjectURL(url); // 釋放記憶體

      canvas.toBlob(
        blob => resolve(new File([blob], file.name, { type: 'image/jpeg' })),
        'image/jpeg',
        0.85 // JPEG 品質 85%，兼顧清晰度與檔案大小
      );
    };
    img.src = url;
  });
}

// 整合入 prepareFileForUpload：
async function prepareFileForUpload(file) {
  const IMAGE_TYPES = ['image/jpeg', 'image/png', 'image/gif'];
  // 1. 圖片先壓縮
  const processedFile = IMAGE_TYPES.includes(file.type)
    ? await _compressImage(file)
    : file;
  // 2. 壓縮後再做大小檢查
  if (processedFile.size > 8 * 1024 * 1024) {
    throw new Error(`檔案仍超過 8MB（${(processedFile.size/1024/1024).toFixed(1)}MB），請手動壓縮後再上傳。`);
  }
  // 3. 轉 Base64
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onload = e => resolve({
      base64: e.target.result.split(',')[1],
      mimeType: processedFile.type,
      fileName: processedFile.name
    });
    reader.onerror = reject;
    reader.readAsDataURL(processedFile);
  });
}
```

## 四之二、效能與併發防護設計

### 4-2.1 寫入併發保護（LockService）

全校教師可能在某場大型研習結束後「同時」湧入登錄，多人同時執行 `submitRecord` 會導致 `TRAINING_RECORD` 工作表的「一次讀取 → setValues 寫回」互相覆蓋。**`Record.gs` 的 `submitRecord` 與 `Review.gs` 的 `reviewRecord` 均須包裝 LockService。**

```javascript
// Record.gs — submitRecord 核心骨架
function submitRecord(userId, body) {
  const lock = LockService.getScriptLock();
  try {
    lock.waitLock(10000); // 最多等 10 秒，逾時拋錯
  } catch (e) {
    throw new Error('系統正忙，請稍後再試（約 10 秒後重新送出）。');
  }
  try {
    const sheet   = _getRecordSheet();
    const allRows = sheet.getDataRange().getValues(); // 一次讀取全表
    // ... 在記憶體中建立新紀錄列，push 至 allRows
    sheet.clearContents();
    sheet.getRange(1, 1, allRows.length, allRows[0].length).setValues(allRows);
  } finally {
    lock.releaseLock(); // 無論成功失敗都必須釋放
  }
}
```

> ⚠️ `waitLock(5000)` 對寫入頻繁的情境稍嫌太短，研習系統改用 **10000ms（10 秒）**。若仍逾時，回傳友善提示而非系統錯誤。

### 4-2.2 批次審核效能（全表讀寫一次）

`reviewRecord` 支援單筆或批次審核。**嚴禁逐列呼叫 setValue**。標準實作模式：

```javascript
// Review.gs — reviewRecord（支援批次）
function reviewRecord(reviewerId, body) {
  // body.records: [{ recordId, status, reviewNote }, ...]
  const lock = LockService.getScriptLock();
  lock.waitLock(10000);
  try {
    const sheet   = _getRecordSheet();
    const data    = sheet.getDataRange().getValues(); // ① 一次讀取全表（含標題列）
    const headers = data[0];
    const idCol   = headers.indexOf('recordId');      // 動態定位欄位索引
    const statusCol   = headers.indexOf('status');
    const reviewerCol = headers.indexOf('reviewedBy');
    const noteCol     = headers.indexOf('reviewNote');
    const atCol       = headers.indexOf('reviewedAt');

    // 建立待審 recordId 的 Hash Map，O(1) 查詢
    const updateMap = {};
    body.records.forEach(r => { updateMap[r.recordId] = r; });

    // ② 記憶體中批次修改
    for (let i = 1; i < data.length; i++) {
      const row = data[i];
      const update = updateMap[row[idCol]];
      if (!update) continue;
      row[statusCol]   = update.status;
      row[reviewerCol] = reviewerId;
      row[noteCol]     = update.reviewNote || '';
      row[atCol]       = new Date().toISOString();
    }

    // ③ 一次寫回全表
    sheet.clearContents();
    sheet.getRange(1, 1, data.length, data[0].length).setValues(data);
  } finally {
    lock.releaseLock();
  }
}
```

### 4-2.3 孤兒檔案防護（重新送出沿用舊檔案）

**問題情境**：教師因研習名稱填錯被退件，重新送出時上傳了同一份 PDF，導致 Google Drive 個人資料夾中出現重複檔案，浪費空間且難以管理。

**設計方案**：前端「重新送出」入口提供兩個選項：

```
重新送出研習證明
  ○ 沿用上次上傳的證明（[原始檔名] · 上傳於 YYYY/MM/DD）
  ○ 重新上傳新檔案  [選擇檔案]
```

- 選擇「沿用」時：前端送出 `{ reuseFileId: "原始 fileId", fileName: "原始 fileName" }`，不含 base64
- 後端邏輯：`submitRecord` 偵測到 `reuseFileId` 存在時，跳過 `_saveFileToDrive()`，直接將 `reuseFileId` 與 `fileName` 寫入新的 `TRAINING_RECORD` 列
- **前提驗證**：後端需確認該 `fileId` 確實屬於該 `userId`（避免偽造他人 fileId）

```javascript
// Record.gs — submitRecord 中的 reuseFileId 分支
function submitRecord(userId, body) {
  // ...（LockService 包裝省略，見 4-2.1）

  let fileId, fileName;

  if (body.reuseFileId) {
    // 驗證舊 fileId 確實屬於此教師
    const prevRecord = _findRecordByFileId(allRows, headers, body.reuseFileId, userId);
    if (!prevRecord) throw new Error('找不到原始檔案，請重新上傳。');
    fileId   = body.reuseFileId;
    fileName = body.fileName;
  } else {
    // 正常上傳流程
    fileId = _saveFileToDrive(userId, body.base64, body.mimeType, body.fileName, body.trainingDate);
    fileName = body.fileName;
  }
  // ... 寫入新 recordId
}
```

## 四之三、通知系統設計（Notify.gs）

### 背景與技術基礎

研習管理的核心痛點：管理者無法主動掌握哪些教師尚未完成必修研習。透過 GAS 內建的 `MailApp.sendEmail()` 與教職員名冊已有的學校 Gmail 欄位，可建立主動通知機制，零外部服務依賴。

| 項目 | 說明 |
|---|---|
| **寄件者** | GAS 腳本擁有者的學校 Gmail（部署後固定） |
| **收件者 email 來源** | `SchoolPortalLib.getUser(txxxx).email`（STAFF.email，格式 `@xxx.edu.tw`） |
| **每日配額** | Google Workspace 帳號：**1,500 封/日**（全校一輪通知不超過 200 封，綽綽有餘） |
| **新增 OAuth Scope** | `appsscript.json` 加入 `https://www.googleapis.com/auth/script.send_mail` |

---

### 三種通知情境

| # | 情境 | 觸發條件 | 收件者 | 觸發頻率 |
|---|---|---|---|---|
| **N1** | 到期前 7 天提醒 | 必修課程距 `endDate` ≤ 7 天，且教師無 PENDING / APPROVED 紀錄 | 教師本人 | 每週一 07:00 |
| **N2** | 逾期未完成通知 | 必修課程已過 `endDate`，且仍無 APPROVED 紀錄 | 教師本人 ＋ **所屬處室的 training_admin 管理者** | 每日 07:00 |
| **N3** | 待審逾時提醒 | PENDING 紀錄超過 **3 天**未被審核 | 教師所屬處室的 training_admin 管理者 | 每日 07:00 |

> **N2 處室管理者對應邏輯**：從 `Hub.UserStatusCache` 讀取教師的 `department`，比對同處室且 `systemAccess.training_admin = true` 的在職管理者，取得其 email。若該處室無管理者，則僅寄給教師本人並在 AuditLog 記錄警告。

### 防重複寄送（Anti-Spam）

同一名教師針對同一門課，同一天內最多寄送一封通知，以 `CacheService` 防止定時器重複觸發時重複寄信：

```javascript
// Notify.gs
const _notifKey = (type, userId, catalogId) => {
  const d = new Date();
  return `notif_${type}_${userId}_${catalogId}_${d.getFullYear()}${d.getMonth()}${d.getDate()}`;
};

function _hasNotifiedToday(type, userId, catalogId) {
  return !!CacheService.getScriptCache().get(_notifKey(type, userId, catalogId));
}

function _markNotified(type, userId, catalogId) {
  // TTL 86400 秒（24 小時），隔日自動清除
  CacheService.getScriptCache().put(_notifKey(type, userId, catalogId), '1', 86400);
}
```

### 核心函式架構

```javascript
// Notify.gs — 主入口（由定時觸發器與手動 API 共用）
function checkAndNotifyOverdue() {
  const today    = new Date();
  const catalog  = parseSheetData(_getCatalogSheet())
                     .filter(c => c.status === 'ACTIVE' && c.isRequired === true);
  const records  = parseSheetData(_getRecordSheet());
  const teachers = _getActiveTeachers(); // 從 Hub.UserStatusCache 取在職教師清單

  // 建立 Hash Map：「userId_catalogId」→ 最高狀態（APPROVED > PENDING > REJECTED）
  const statusMap = {};
  records.forEach(r => {
    const key = `${r.userId}_${r.catalogId}`;
    if (!statusMap[key] || r.status === 'APPROVED') statusMap[key] = r.status;
  });

  catalog.forEach(course => {
    // ⚠️ 日期解析規範：使用數字拆分，禁用 new Date("YYYY/M/D")
    const [y, m, d]  = course.endDate.split('/').map(Number);
    const endDate    = new Date(y, m - 1, d);
    const daysLeft   = Math.ceil((endDate - today) / 86400000);

    teachers.forEach(teacher => {
      const key       = `${teacher.userId}_${course.catalogId}`;
      const topStatus = statusMap[key];
      if (topStatus === 'APPROVED') return; // 已完成，略過

      if (daysLeft > 0 && daysLeft <= 7) {
        // N1：即將到期，教師本人
        if (!_hasNotifiedToday('N1', teacher.userId, course.catalogId)) {
          _sendN1Reminder(teacher, course, daysLeft);
          _markNotified('N1', teacher.userId, course.catalogId);
        }
      } else if (daysLeft <= 0) {
        // N2：已逾期，教師本人 + 處室管理者
        if (!_hasNotifiedToday('N2', teacher.userId, course.catalogId)) {
          const adminEmails = _getAdminEmailsByDept(teacher.department);
          _sendN2Overdue(teacher, course, adminEmails);
          _markNotified('N2', teacher.userId, course.catalogId);
        }
      }
    });
  });

  // N3：PENDING 超過 3 天，通知對應處室管理者
  _checkPendingOverdue(records, teachers, today);
}

// 取得指定處室的 training_admin 管理者 email 清單
function _getAdminEmailsByDept(department) {
  const hub     = SpreadsheetApp.openById(HUB_SPREADSHEET_ID);
  const data    = hub.getSheetByName('UserStatusCache').getDataRange().getValues();
  const headers = data[0];
  const deptCol   = headers.indexOf('department');
  const accessCol = headers.indexOf('systemAccess');
  const emailCol  = headers.indexOf('email');
  const statusCol = headers.indexOf('status');
  const ACTIVE = ['在職', '轉調'];

  return data.slice(1)
    .filter(row => {
      if (!ACTIVE.includes(row[statusCol])) return false;
      if (row[deptCol] !== department) return false;
      try { return JSON.parse(row[accessCol] || '{}').training_admin === true; }
      catch { return false; }
    })
    .map(row => row[emailCol])
    .filter(Boolean); // 過濾空值
}
```

// N1 — 即將到期（寄給教師）
function _sendN1Reminder(teacher, course, daysLeft) {
  MailApp.sendEmail({
    to: teacher.email,
    subject: `【研習提醒】${course.title} 距截止日僅剩 ${daysLeft} 天`,
    htmlBody: `
      <p>${teacher.name} 老師您好，</p>
      <p>您尚未完成必修研習 <strong>「${course.title}」</strong>（${course.hours} 小時）。</p>
      <p>截止日期：<strong>${course.endDate}</strong>（還有 ${daysLeft} 天）</p>
      <p>請盡快登錄研習紀錄並上傳研習證明。</p>
      <p style="color:#888;font-size:12px;">此信由研習登錄系統自動寄送，請勿直接回覆。</p>
    `
  });
  SchoolPortalLib.logAction(teacher.userId, 'NOTIFY_N1', course.catalogId);
}

// N2 — 已逾期（寄給教師 + 管理者）
function _sendN2Overdue(teacher, course, adminEmails) {
  const teacherBody = `
    <p>${teacher.name} 老師您好，</p>
    <p>必修研習 <strong>「${course.title}」</strong> 已於 ${course.endDate} 截止，
       但您尚無通過審核的登錄紀錄。</p>
    <p>請盡快聯繫所屬處室管理者說明情況，或補登研習紀錄。</p>
    <p style="color:#888;font-size:12px;">此信由研習登錄系統自動寄送，請勿直接回覆。</p>
  `;
  MailApp.sendEmail({ to: teacher.email,
    subject: `【研習逾期】${course.title} 尚未完成，請盡快處理`, htmlBody: teacherBody });

  if (adminEmails.length) {
    const adminBody = `
      <p>管理者您好，</p>
      <p>以下教師的必修研習 <strong>「${course.title}」</strong>（截止 ${course.endDate}）
         尚無通過審核的紀錄，請追蹤確認：</p>
      <ul><li>${teacher.name}（${teacher.userId}）</li></ul>
      <p style="color:#888;font-size:12px;">此信由研習登錄系統自動寄送，請勿直接回覆。</p>
    `;
    MailApp.sendEmail({ to: adminEmails.join(','),
      subject: `【研習管理】${teacher.department} ${teacher.name} 必修研習逾期未完成`,
      htmlBody: adminBody });
  }
  SchoolPortalLib.logAction(teacher.userId, 'NOTIFY_N2', course.catalogId);
}

## 四之一、與整合平台的資料模型對應

本系統在跨系統 ER 圖中的定位：

| 實體 | 本系統欄位 | 對應系統 | 對應欄位 |
|---|---|---|---|
| **使用者** | `userId (txxxx)` | 教職員名冊 | `userId (txxxx)`（主鍵） |
| **使用者部門** | 不儲存，動態查詢 | Hub.UserStatusCache | `department` |
| **研習課程** | `catalogId`、`title`、`hours` | 本系統獨立維護（無對應資產） | ─ |
| **登錄紀錄** | `recordId`、`trainingDate`、`status` | ─（不涉及設備借用） | ─ |
| **研習統計** | `approvedHours`（同步至 Hub） | 主門戶儀表板 | Hub.TrainingStats |

**本系統與其他子系統的關係**：
- **唯讀依賴** Hub.UserStatusCache → 取得教師部門資訊（供統計彙整使用）
- **單向寫入** Hub.TrainingStats → 提供儀表板 KPI 資料來源
- **無設備借用關聯**：研習登錄不涉及資產/設備，不接入 Hub.EquipmentRegistry
- **身分驗證**：透過 SchoolPortalLib 共用，不直接讀取教職員名冊試算表

### 5.2 REJECTED 紀錄操作
- 顯示退件原因文字（必顯示，不可隱藏）
- 提供「重新送出」按鈕，點後跳至送出頁並預填原有課程資料
- 原 REJECTED 紀錄保留（不刪除），新送出建立新 recordId
- **沿用舊檔案選項**（避免孤兒檔案）：送出頁面預設顯示「沿用上次上傳的證明」與「重新上傳」二選一；若退件原因是課程填錯而非證明有問題，教師可選擇沿用，後端直接複用舊 fileId，不重新上傳至 Drive

## 九、開發規範提醒（每次開工確認）

- 輸出語言：繁體中文（台灣慣用語）
- 試算表讀寫：「一次讀取 → 記憶體處理 → 一次 setValues 寫入」，禁止在迴圈內 setValue
- **前端圖片壓縮（必做）**：JPG/PNG 上傳前必須以 Canvas API 壓縮至 2MB 以下再轉 Base64，防止手機照片逾時（見 Section 4.4.1）
- **LockService 必包（必做）**：`submitRecord`、`reviewRecord` 等所有寫入 TRAINING_RECORD 的函式，均需使用 `LockService.getScriptLock().waitLock(10000)` 保護，並在 `finally` 中 `releaseLock()`（見 Section 四之二）
- 前端 API：禁止 `fetch()`，全部使用 `google.script.run` 封裝的 `api()` 函式
- Inline event：所有前端函式須在 `<script>` 末尾掛載至 `window.xxx = xxx`
- CSS 斷點：桌面端用 `min-width: 1024px`（GAS 沙箱 viewport 問題，勿用 768px）
- 日期建構：一律用 `new Date(year, monthIndex, day)`，禁用 `new Date("YYYY/M/D")`
- 跨系統查詢：透過 Hub 試算表中介，禁止直接 `openById` 其他子系統試算表
- 個資保護：證明檔案僅管理者可讀寫，本人唯讀；禁止公開連結
- REJECTED 紀錄：保留歷史，不覆蓋；重新送出建立新 recordId；前端提供「沿用舊檔案」選項避免孤兒檔案
- **MailApp 每日配額（必知）**：Google Workspace 帳號上限 1,500 封/日。`checkAndNotifyOverdue()` 每日執行一次，加上 CacheService 防重複，正常情境不會超標。若未來全校齊發，預估 < 200 封/日。禁止在迴圈內無節制呼叫 `MailApp.sendEmail()`
- **通知 email 來源**：一律透過 `SchoolPortalLib.getUser(txxxx).email` 取得，禁止在試算表中重複儲存 email 欄位（以教職員名冊為唯一來源）

## 十一、系統設計哲學（人工審核原則）

> 本章節確立系統邊界，防止過度自動化導致認列糾紛，亦是對行政人員的設計承諾。

### 11.1 核心原則：行政人員保有最終認列決定權

本系統的定位是「**輔助收件 + 清單管理 + 提醒通知**」，而非「自動認列系統」。不論研習時數、日期、對象是否符合規定，**系統一律不自動判斷是否合格**，所有研習紀錄的認列皆由行政人員**人工確認後點擊核准**。

| 功能 | 系統角色 | 決策者 |
|---|---|---|
| 上傳研習證明 | 接收並儲存檔案 | 教師 |
| 時數是否符合規定 | **不自動判斷** | 行政人員人工確認 |
| 研習對象是否符合 | **不自動判斷** | 行政人員人工確認 |
| 日期是否在有效期間內 | **不自動判斷** | 行政人員人工確認 |
| 核准 / 退件 | 執行並記錄結果 | 行政人員點擊 |

### 11.2 自動化功能的定位

系統現有的自動化功能（通知、統計同步）均屬「**輔助工具**」，不參與認列決策：

| 自動化功能 | 定位 | 說明 |
|---|---|---|
| N1/N2 到期通知 | 提醒工具 | 提醒教師注意截止日，不代表系統自動判定「未通過」 |
| N3 待審提醒 | 工作提醒 | 提醒管理者處理積壓，不自動核准 |
| Hub.TrainingStats 同步 | 統計工具 | 彙整已核准時數供儀表板參考，非認列依據 |
| `isRequired` 必修標記 | 公告工具 | 協助教師識別必修課程，不影響審核流程 |

### 11.3 不適合本系統的功能（明確排除）

以下功能在本系統**刻意不實作**，確保行政彈性：
- ❌ 自動比對教師已完成時數是否達標
- ❌ 自動阻擋重複上傳同一研習的功能（允許補件）
- ❌ 強制要求研習對象必須符合才能上傳
- ❌ 超過截止日自動將紀錄標記為「逾期不予認列」

> 若未來有需求，可在**新的獨立模組**（如「研習時數統計報表」）中實作，而非修改核心審核流程，確保現有功能不受影響。
